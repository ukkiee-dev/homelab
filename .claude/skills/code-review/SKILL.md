# Code Review — 종합 리뷰 오케스트레이터

4개 전문 에이전트로 코드를 병렬 감사하고, 교차 관심사를 식별하여 통합 보고서를 산출한다.

## 실행 모드: 서브 에이전트 (Sub-agent, Fan-out/Fan-in)

4명의 리뷰어가 독립적으로 감사하고, 오케스트레이터가 중복 제거 + 교차 관심사 식별 + 통합한다.

## 에이전트 풀

| 에이전트 | 모델 | 리뷰 차원 |
|---------|------|----------|
| `arch-reviewer` | opus | 모듈 구조, 의존성, 설계 패턴, 확장성 |
| `security-reviewer` | opus | OWASP Top 10, 인증/인가, 인젝션, 시크릿 |
| `perf-reviewer` | opus | 복잡도, N+1, 메모리, 동시성, 캐싱 |
| `style-reviewer` | opus | 가독성, 네이밍, DRY, 타입, 테스트 |

## 워크플로우

### Phase 1: 범위 분석

1. **입력 파악**: 사용자가 제공한 리뷰 대상을 식별한다
   - 파일 경로 → 해당 파일 직접 리뷰
   - git diff / 브랜치 → 변경된 파일 목록 추출
   - "전체 프로젝트" → 핵심 파일 식별 (진입점, 라우터, 모델, 서비스 계층)
2. **언어/프레임워크 감지**: 프로젝트의 기술 스택을 파악한다
3. **리뷰 범위 저장**: `_workspace/00_input/review-scope.md`에 기록

```markdown
# 리뷰 범위
- **대상**: [파일 목록 또는 diff 범위]
- **언어/프레임워크**: [감지된 스택]
- **코드 규모**: [파일 수, 라인 수 추정]
- **리뷰 깊이**: full / focused (변경분만)
```

### Phase 2: 병렬 리뷰 (Fan-out)

4개 에이전트를 **동시에** 스폰한다. 모든 호출에 `model: "opus"` 필수.

각 에이전트 프롬프트에 포함할 내용:
- 리뷰 대상 파일 경로 또는 diff
- 언어/프레임워크 정보
- 에이전트 정의 파일(`.claude/agents/{name}.md`)의 출력 형식을 따르라는 지시
- 결과를 `_workspace/` 지정 파일에 저장하라는 지시

```
Agent(arch-reviewer, run_in_background=true, prompt: "...")
Agent(security-reviewer, run_in_background=true, prompt: "...")
Agent(perf-reviewer, run_in_background=true, prompt: "...")
Agent(style-reviewer, run_in_background=true, prompt: "...")
```

### Phase 3: 통합 보고서 작성

4개 결과를 읽고 통합한다. 이 단계의 핵심은 단순 병합이 아니라 **교차 분석**이다.

#### 3-1. 중복 제거
동일 파일:라인을 여러 리뷰어가 지적한 경우 하나로 합치고, 복수 관점을 병기한다.
예: "이 함수가 길다(스타일) + 이 함수에 SQL 인젝션이 있다(보안)" → 하나의 항목에 두 태그

#### 3-2. 교차 관심사 식별
서로 다른 차원의 발견이 연결되는 패턴을 찾는다:
- 아키텍처 결함 → 보안 취약점 (예: 인증 모듈이 분리되지 않아 우회 경로 존재)
- 성능 병목 → 아키텍처 변경 필요 (예: N+1 해결에 캐싱 레이어 도입)
- 스타일 이슈 → 버그 위험 (예: 에러 처리 누락 → 미처리 예외)

#### 3-3. 심각도 통합 정렬
모든 발견을 통합 심각도로 재정렬한다:
- **Critical**: 즉시 수정 (보안 취약점, 데이터 손실 위험, 프로덕션 장애)
- **Warning**: 계획적 수정 (성능 병목, 설계 결함, 유지보수 위험)
- **Info**: 개선 권고 (스타일, 미세 최적화, 베스트 프랙티스)

교차 관심사가 있는 항목은 심각도를 한 단계 올린다.

#### 3-4. 보고서 생성
`_workspace/05_integrated_report.md`에 저장하고, 핵심 내용을 사용자에게 직접 출력한다.

보고서 형식은 `references/review-checklists.md`의 "통합 보고서 템플릿" 참조.

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| 리뷰어 1개 실패 | 나머지 3개로 보고서 작성, 누락 차원 명시 |
| 리뷰어 2개+ 실패 | 사용자에게 알리고 부분 결과 제공 |
| 프로젝트가 너무 큼 | 핵심 파일(진입점, 라우터, 모델)만 리뷰. 전체 리뷰는 모듈 단위 분할 제안 |
| 언어 미지원 | 일반 원칙(SOLID, OWASP)으로만 리뷰, 언어 특수 패턴은 유보 |

## 테스트 시나리오

### 정상 흐름: PR 리뷰
1. **입력**: "이 PR 리뷰해줘" + git diff 또는 파일 목록
2. 범위 분석 → 변경 파일 10개, TypeScript + React
3. 4개 리뷰어 병렬 스폰
4. arch: 컴포넌트 구조 검토 / security: XSS, 인증 검토 / perf: 리렌더링, 번들 검토 / style: 네이밍, 타입 검토
5. 통합: 중복 제거 3건, 교차 관심사 1건(에러 처리 누락→보안) 발견
6. Critical 2, Warning 5, Info 8 → 통합 보고서 출력

### 에러 흐름: 부분 리뷰
1. **입력**: "이 Python 코드 리뷰해줘" + 파일 3개
2. 4개 리뷰어 스폰 → security-reviewer 타임아웃
3. 나머지 3개 결과로 보고서 작성
4. "보안 리뷰 미완료 — 별도 보안 감사 권장" 경고를 보고서 상단에 명시
